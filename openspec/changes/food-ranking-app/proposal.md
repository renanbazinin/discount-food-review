## Why

The user wants a simple, pretty, Hebrew/RTL web app to rank food dishes they can order from a small set of 10bis restaurants. Existing food apps optimize for ordering or discovery, not for the act of deciding "do I like A or B more." A focused pairwise-comparison app produces a personal leaderboard that doubles as a decision aid before the next order, and makes the ranking process itself feel like a game instead of a chore.

## What Changes

- Introduce a static dataset of main-course dishes scraped from a configurable list of 10bis restaurants (initial seed: 5 restaurants, 79 dishes, already captured under `data/`).
- Introduce a standalone scrape tool that reads a restaurant-id config file, fetches menus from the 10bis public `NextApi/getRestaurantMenu` endpoint, classifies main courses via a Hebrew keyword allowlist, duplicates טלבנק dishes into two variants (original + cleaned), downloads dish images, and writes `data/restaurants.json`.
- Introduce a phone-first, desktop-friendly web app with two screens: a pairwise comparison screen (primary interaction) and a ranking list screen (secondary view).
- Introduce an Elo-based ranking system, seeded from the scraped `dishPopularityScore` signal so the first comparisons already feel reasonable.
- Introduce a storage abstraction (`RatingsStore` interface) backed by an append-only match-event log, with a LocalStorage adapter used today. A future MongoDB adapter will slot in without UI changes.
- Hebrew/RTL is first-class: `dir="rtl"`, logical CSS properties, a Hebrew webfont, and no language toggle.
- Twin filter: never pair two dishes that share the same root id (so a טלבנק variant is never matched against its non-טלבנק twin).

## Capabilities

### New Capabilities

- `menu-scraping`: fetches 10bis restaurant menus by id, classifies categories/dishes as main courses, splits טלבנק dishes into variants, downloads images, and emits a versioned static JSON dataset consumed by the app.
- `dish-catalog`: the runtime model of restaurants and dishes loaded by the app from the static JSON dataset, including lookup, filtering, and twin-pair detection.
- `pairwise-ranking`: presents two dishes at a time, records the user's pick as a match event, updates Elo ratings, and selects the next pair with a bias toward under-sampled and close-rated dishes.
- `ratings-store`: persistence abstraction for match events and derived state, with a LocalStorage adapter now and a MongoDB adapter pluggable later without touching UI code.
- `ranking-view`: renders the current leaderboard derived from match events, separates "confident" from "under-sampled" dishes, and provides export/import of the event log.

## Impact

- New top-level directories: `app/` (web app source), `scripts/` (scrape tool), `data/` (static dataset, already partially populated).
- New runtime dependencies: a web framework (SvelteKit recommended) and Tailwind CSS for RTL-friendly styling. The scrape tool is a standalone script with no runtime dependency on the app.
- No backend, no accounts, no network calls at runtime. All app state lives in `localStorage` behind the `RatingsStore` interface.
- The existing `data/restaurants.json` and `data/images/` produced during exploration are the v1 seed and will be regenerated by the new scrape tool on first run.
- Future MongoDB integration is explicitly out of scope for this change but is a first-class design constraint: the storage interface and event-log model must make the later swap trivial.
